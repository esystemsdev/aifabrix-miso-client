# Plan 59.3 — Browser User-Token Storage Hard-Stop

## Goal
Achieve full `192.2` browser-token alignment in `aifabrix-miso-client` by enforcing that browser runtime paths have no JS-readable operational source for user token material, while preserving cookie/session continuity, device refresh boundaries, and 59.2 M2M runtime-memory-only behavior.

## Goal Completion Criteria
- **G1 User-token runtime hard-stop**: active browser runtime paths have no JS-readable operational source for user token material (including localStorage/sessionStorage).
- **G2 No fallback reintroduction**: no compatibility paths re-enable browser token persistence (user or M2M).
- **G3 Boundary continuity**: strict browser-session behavior (59.1), M2M runtime-memory behavior (59.2), and device refresh boundaries remain non-regressed.
- **G4 Closure evidence**: fail-closed runtime snapshots, response samples, tests/docs/changelog, and ordered validation logs prove final behavior.

## Goal-to-Section Alignment
- **Scope** includes only work that drives `G1-G4` closure.
- **Implementation Strategy** is sequenced to remove runtime persistence first, then prove boundary non-regression.
- **Expected Automated Tests** and **Required Evidence** are direct proof obligations for `G1-G4`.
- **Validation Sequence** and **Definition of Done** close the Goal only when all criteria are simultaneously satisfied.

## Contract Baseline (59.3)
- Browser JS must not use any JS-readable user-token operational source in browser runtime paths.
- Browser auth continuity must rely on cookie/session semantics.
- Refresh token remains server-only and is never exposed via browser-readable payloads in strict browser paths.
- M2M client token remains runtime-memory only (no browser persistent storage).
- No backward-compatibility fallback layers are allowed for 59.3 target behavior.

## Allowed / Forbidden Matrix
- **Allowed**
  - Cookie/session restore-refresh continuity mechanisms for browser auth.
  - Runtime/config token sources only for non-browser/device or explicitly non-browser paths.
  - Non-token config caching (for example `miso:dataclient-config`).
- **Forbidden**
  - Any JS-readable user-token operational source in active browser runtime paths.
  - Operational read/write dependency on `miso_token` / `miso_token_expires_at` in active browser runtime paths.
  - Treating legacy token keys as required inputs for request authorization flow.
  - Reintroduction of `miso:client-token*` persistence or fallback behavior.
  - Any browser-readable refresh-token exposure introduced while removing persistence paths.

## Scope
### In scope
- Remove JS-readable user-token operational sources from active browser request/auth flows.
- Remove browser runtime dependency on persisted user token keys (`miso_token`, `miso_token_expires_at`) as part of the hard-stop.
- Keep browser session helpers cookie-first and non-refresh-token-exposing.
- Preserve device refresh boundary and M2M client-token runtime-memory behavior from 59.2.
- Update tests/docs/changelog to match final no-persistent-browser-token behavior.

### Out of scope
- Controller (`aifabrix-miso`) backend contract changes.
- Python SDK (`aifabrix-miso-client-python`) implementation.
- Release/tag creation.

## Implementation Strategy
1. Freeze contract delta for 59.3
   - Explicitly mark persistent browser user-token storage as forbidden for active runtime paths.
   - Define allowed sources with explicit path boundaries:
     - browser paths: cookie/session restore-refresh continuity only (no JS-readable user-token operational source);
     - non-browser/device paths only: runtime-memory or explicit runtime-config token sources where contract requires them.

2. Remove browser persistence usage in active runtime
   - Remove operational reads/writes of `miso_token` and `miso_token_expires_at` in all active browser runtime paths (no exceptions).
   - Ensure browser auth continuity uses cookie/session semantics rather than JS-readable token sources.
   - Preserve canonical data shapes and existing public method signatures unless strict removal is required.

3. Keep boundary guarantees intact
   - No reintroduction of refresh-token browser persistence.
   - No reintroduction of `miso:client-token*` persistence/fallback (59.2 guardrail).
   - No regression in device refresh route semantics.

4. Align tests to strict no-persistent-browser-token behavior
   - Convert tests that currently assume localStorage user-token persistence into runtime/callback assertions.
   - Add deterministic assertions for:
     - empty localStorage startup;
     - stale legacy key presence ignored operationally;
     - successful auth flow without persisted browser token keys.
   - Keep redirect/auth boundary tests green by switching fixtures to runtime/config token sources where needed.

5. Update documentation and migration guidance
   - Remove any remaining instructions implying browser persistence of user/M2M token material.
   - Add concise migration notes for consumers relying on `miso_token*` localStorage behavior.

# Plan 59.2 — `aifabrix-miso-client` Client-Token Persistence Hard-Stop

## Goal
Remove all browser persistence and localStorage fallback usage for client-token material in `aifabrix-miso-client`, while preserving completed 59.1 strict browser-session hard-cut behavior and existing device/M2M boundaries.  
Result expected from this plan: browser auth/runtime paths no longer read/write `miso:client-token` keys in browser storage, docs/migration notes are updated to runtime-memory-only semantics, and regression tests plus validation gates are green.  
Reference only: [192.2 goal alignment plan](/workspace/aifabrix-miso/.cursor/plans/192.2_goal_alignment_85d5f32e.plan.md).

## Goal Completion Criteria
The Goal is considered fully achieved only when all four criteria below are true at the same time:
- **G1 Storage hard-stop**: active browser runtime paths do not persist `miso:client-token` / `miso:client-token-expires-at`.
- **G2 Fallback hard-stop**: active browser runtime paths do not use localStorage fallback for client-token operational behavior.
- **G3 Contract continuity**: 59.1 strict browser-session hard-cut guardrails remain intact (no fallback reintroduction, no boundary regressions).
- **G4 Closure evidence**: docs/tests/validation evidence is synchronized and all required gates pass in strict order.

## Verbatim Source Contract Statement
The following contract statement is copied exactly from the source cross-repo plan Goal:
- `refresh token` is server-only.
- `user access token` is in HttpOnly cookie (browser JS does not read/store it).
- `M2M client token` stays runtime-memory only (no persistent browser storage).

## Contract Baseline (for this repo)
- Browser JS must not persist M2M client-token material in `localStorage`/`sessionStorage`.
- Browser runtime must not use `localStorage` client-token fallback for operational behavior.
- Browser session/auth strict parsing from 59.1 remains intact (no fallback reintroduction).
- Device refresh and service/M2M boundaries stay unchanged.
- No backward-compatibility fallback layers are allowed for the 59.2 target behavior.

## Allowed / Forbidden Matrix
- **Allowed**
  - Runtime/config/callback client-token sourcing without browser persistence.
  - Non-token browser config caching that does not carry client-token material.
  - 59.1 strict browser/session behavior preserved as-is.
- **Forbidden**
  - Reading/writing `miso:client-token` or `miso:client-token-expires-at` in active browser runtime paths.
  - localStorage fallback as an operational source for client-token behavior.
  - Reintroducing browser/session fallback parsing or refresh-token persistence previously removed by 59.1.
  - Silent drift between implementation, tests, docs, and changelog.

## Known Baseline Contradictions
- Browser helper/client-token paths currently include `miso:client-token` and `miso:client-token-expires-at` persistence behavior that conflicts with 192.2 runtime-memory-only intent for M2M token material.
- `ApplicationContextService` currently contains browser localStorage fallback logic for client-token extraction, which conflicts with strict runtime-memory-only behavior.
- Existing docs still contain localStorage guidance for client-token key usage and must be corrected to avoid contract drift.

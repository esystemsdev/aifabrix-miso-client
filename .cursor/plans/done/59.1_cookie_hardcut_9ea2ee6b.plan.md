## Implementation Steps
1. **Freeze hard-cut SDK contract**
   - Treat browser auth as cookie/session-driven only.
   - Treat browser `refreshToken` request payload as non-required/ignored.
2. **Align API/type surfaces**
   - Ensure browser refresh calls use `/api/v1/auth/refresh` without JSON body requirements.
   - Keep response typing stable and camelCase for consumers.
3. **Align browser runtime behavior**
   - Remove browser refresh-token storage/hydration dependency.
   - Keep deterministic auth cleanup and session-failure handling.
4. **Protect boundaries**
   - Keep device refresh contract body-based and unchanged.
   - Keep M2M `x-client-token` behavior unchanged.
   - Add explicit guard tests to prevent browser/device boundary regressions.
5. **Sync docs and release notes**
   - Update SDK docs (`README`, `docs/*`) and `CHANGELOG.md` with hard-cut semantics.
   - Include migration notes for consumers affected by browser token persistence assumptions.
6. **Validate and close**
   - Run silent-first validation sequence.
   - Re-run final verification loop until no new issues remain.

## Implementation Guardrails
- Do not change device refresh request contract (`/api/v1/auth/login/device/refresh` body `refreshToken`).
- Do not modify M2M token acquisition/usage semantics.
- Do not introduce fallback aliases that re-enable legacy browser token persistence.
- Keep changes scoped to `aifabrix-miso-client`; cross-repo edits are out of scope for this plan.
- No multi-shape parsing fallback in strict browser/session flow (`data` envelope vs flat payload must be canonicalized to one contract).
- No token-name alias fallback in strict browser/session flow (`accessToken` only, no `token`/`authToken` aliases).

## Expected Automated Tests
- Browser refresh calls `/api/v1/auth/refresh` without JSON request body dependency.
- Browser runtime/session helpers do not persist or hydrate refresh token in browser flow.
- Device refresh path `/api/v1/auth/login/device/refresh` keeps required body `refreshToken`.
- Auth service refresh mapping remains deterministic for partial/malformed response shapes.
- M2M `x-client-token` behavior remains non-regressed and separate from browser user-session flow.
- [EDGE] Browser refresh callback present but returns empty/partial payload: deterministic auth behavior with no token leakage.
- [EDGE] Browser session cleanup after failed restore/refresh remains idempotent across repeated failures.
- [EDGE] Browser/session strict parser rejects non-canonical response shape (no nested/flat dual parsing fallback).
- [EDGE] Browser/session strict parser rejects token alias fields when canonical field is required (`accessToken` only in strict path).
